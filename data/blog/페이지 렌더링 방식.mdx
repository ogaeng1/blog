#### 렌더링이란 무엇일까?

사람들은 하루에 수십 번씩 웹브라우저에 접속합니다. 미리 만들어진 페이지를 불러오는 게 아니라 사용자가 인식하지 못할 속도로 페이지가 실시간으로 만들어집니다. 실시간으로 웹사이트가 만들어지는 과정을 **렌더링**이라고 말할 수 있습니다.

페이지 렌더링 방식에는 다음과 같이 3가지로 나눌 수 있습니다.

1. SSR
2. CSR
3. SSG

> ### SSR(Server-side Rendering)

SSR은 서버에서 페이지를 렌더링하는 방식을 말합니다.

SSR의 동작 방식은 다음과 같습니다.

![](/static/images/ssr.png)

1. 클라이언트(브라우저)가 특정 페이지를 요청합니다.
2. 서버에서는 해당 페이지에 필요한 데이터를 포함한 HTML 파일을 클라이언트에 전달합니다.
3. 클라이언트에서는 HTML 파일을 파싱 하면서 필요한 정적 리소스(CSS, 이미지 등)와 자바스크립트 파일을 다운로드하며, 페이지를 렌더링 합니다.
4. 클라이언트에서 다운로드한 자바스크립트 파일을 실행 합니다.
5. 페이지 인터렉션이 가능합니다.

#### SSR 특징은 무엇이 있을까?

1. **최초 페이지 로딩 시간이 짧다**

   - 요청한 페이지에 필요한 리소스만 다운로드 하기 때문에 렌더링에 필요한 시간이 짧습니다.

2. **자바스크립트 실행이 완료될 때까지 인터렉션이 불가능**

   - SSR 방식은 사용자가 더 빨리 페이지를 조회할 수 있지만 자바스크립트 실행이 완료될 때까지는 페이지 내에서의 상호 작용이 불가능 합니다.

3. **페이지 이동에 걸리는 시간이 CSR 방식보다 더 길다**

   - 페이지를 이동할 때마다 해당 페이지에 필요한 리소스를 다운로드하고 페이지를 다시 생성해야 하므로 CSR 방식에 비해 페이지 이동에 걸리는 시간은 더 깁니다.

4. **검색엔진 최적화(SEO)에 적합**
   - 서버에서 페이지를 생성하므로 검색엔진에서는 빈 HTML 문서가 아닌 완성된 HTML 문서를 수집할 수 있으므로 검색엔진 최적화에 용이합니다.

> ### CSR(Client-sid Rendering)

CSR은 클라이언트(브라우저)에서 페이지를 렌더링 하는 방식을 말합니다.

CSR의 동작 방식은 다음과 같습니다.

![](/static/images/csr.png)

1. 클라이언트가 특정 페이지를 요청합니다.
2. 서버에서는 빈 HTML 문서를 클라이언트에 전달 합니다.
3. 클라이언트가 HTML 문서에 포함된 정적 리소스(CSS, 이미지 등), 번들 자바스크립트 파일을 다운로드 합니다.
4. 클라이언트에서 번들 자바스크립트 파일을 실행 합니다.
5. 페이지 렌더링이 완료된 후 API에서 받아온 데이터로 state를 업데이트하고 페이지를 렌더링 합니다.
6. 페이지 인터렉션이 가능합니다.

#### CSR 특징은 무엇이 있을까?

1. **최초 페이지 로딩 시간이 SSR보다 오래 걸린다**

   - 렌더링에 필요한 리소스와 번들 자바스크립트 파일을 다운로드 한 후 화면을 그리는 데 걸리는 시간이 오래 걸리기 때문에 전체적인 렌더링 완료 시간은 오래 걸리게 됩니다.

2. **페이지 이동에 걸리는 시간이 짧다**

   - 페이지 렌더링에 필요한 정적 리소스와 번들 자바스크립트 파일은 최초 페이지 접근 시에만 다운로드하고, 이동할 페이지에 필요한 부분만 추가로 요청하므로 페이지 이동에 걸리는 시간은 짧습니다.

3. **데이터 응답을 받아올 때까지 빈 화면(혹은 로딩) 표출**
   - CSR 방식은 번들 자바스크립트 실행이 완료된 후, API 응답을 받아오기 전까지는 빈 화면(혹은 별도로 설정한 로딩 화면)이 표출됩니다.

> ### SSG(Static Site Generation)

SSG는 프로젝트 빌드 시 페이지를 사전 렌더링 하는 방식입니다.
미리 만들어 둔 페이지를 클라이언트에게 제공하므로 렌더링 속도가 매우 빠릅니다.
SSR과 마찬가지로 완성된 페이지를 클라이언트에게 제공하기 때문에 SEO에도 친화적입니다.
블로그나 정보성 페이지같이 정적인 데이터를 보여주는 사이트에 적합합니다.
